Доступний акордеон з ARIA та клавіатурною навігацією

Вмикайте/вимикайте секції за допомогою кнопок. Навігація між заголовками: Arrow Up / Arrow Down, перехід до першої/останньої: Home / End, табуляція: Tab.

Доступність в веб-розробці — це практика проєктування вебсайтів, додатків та інструментів, щоб люди з інвалідністю могли їх використовувати. Приблизно 15% населення світу має деяку форму інвалідності, включаючи вади зору, слуху, рухомості та когнітивні порушення.

WCAG 2.2 (Web Content Accessibility Guidelines) — це міжнародний стандарт, який визначає мінімальні рівні доступності контенту. Існує три рівні відповідності: A, AA та AAA, де AA вважається "золотим стандартом" для більшості вебсайтів.

ARIA (Accessible Rich Internet Applications) — це набір атрибутів, які передають дополнительну семантичну інформацію допоміжним технологіям, таким як екранні диктори.

  • aria-expanded: Вказує, чи розгорнута секція (true) чи згорнута (false)
  • aria-controls: Зв'язує кнопку з панеллю, якою вона керує
  • aria-labelledby: Зв'язує панель з її заголовком для синтезу мови
  • aria-hidden: Приховує елементи від дерева доступності

Правильна клавіатурна навігація критично важлива для користувачів, які не можуть використовувати мишку. Акордеон повинен підтримувати:

Клавіша Дія
Tab Рухатися до наступної кнопки (стандартна табуляція)
Shift + Tab Рухатися до попередньої кнопки
Arrow Down Фокус на наступну кнопку (циклічно)
Arrow Up Фокус на попередню кнопку (циклічно)
Home Фокус на першу кнопку
End Фокус на останню кнопку
Enter / Space Перемикання стану панелі (вмикання/вимикання)

Після реалізації акордеона критично важливо перевірити його доступність за допомогою інструментів розробника:

  1. Chrome DevTools Accessibility Tree: Відкрийте DevTools (F12), перейдіть на вкладку Elements → Accessibility, щоб побачити дерево доступності та перевірити, як допоміжні технології бачать вашу сторінку.
  2. Keyboard Testing: Тестуйте лише з клавіатури — переконайтеся, що можна переміщуватися між всіма кнопками та активувати їх без миші.
  3. Screen Reader Testing: Якщо можливо, протестуйте з екранним дикуром (NVDA, JAWS або вбудованим дикуром ОС), щоб переконатися, що контент озвучується правильно.
  4. Lighthouse Audit: Запустіть Chrome Lighthouse для отримання автоматичного звіту про доступність, що грунтується на WCAG рекомендаціях.
  5. Axe DevTools: Встановіть розширення Axe DevTools, щоб отримати детальну перевірку на порушення доступності та рекомендації для розташування.
